# Documentation Authorship

`normal` keeps one small authorship note in the **footer** of each document, so a reader can tell what kind of voice they're reading without a label elbowing into the words.

## Labels

| Label | Meaning |
| --- | --- |
| **User-written** | written directly by the project author |
| **Human/AI-authored** | substantively rooted in direct author wording and intent, with meaningful AI drafting or editing support |
| **Gently user-edited** | rooted in direct author wording and voice, with light structural editing, spelling correction, and only minimal summarization |
| **Agent-written** | written primarily by an AI agent for mechanical, operational, or support clarity |

## How It's Applied

- Each document carries **one** authorship note, in the footer. No labels scattered through the running text.
- The note states the document's overall voice and links back here.
- If a single section genuinely diverges from that overall voice, it earns a small inline anchor — `[1]`, `[2]`, … — resolved in the same footer block. Use these sparingly; minimal visible attribution is the entire point.

## Which Label, Which Doc

- install, command, support, and safety docs can be **Agent-written**
- principle, intent, and product-voice documents (the README, the [Statement](statement.md)) should prefer **User-written**, **Human/AI-authored**, or **Gently user-edited**
- changelog and roadmap entries can be **Agent-written** as long as they stay concrete and honest

Code in this repo should be assumed AI-generated by default unless stated otherwise.
