# Wireframe Proposal Spec B: Compressed Instrument

## Use

This is a prompt-ready wireframe specification for an external tool such as Google AI Studio. It must produce an interactive wireframe, not polished final UI art.

The proposal must follow the governing spec in `docs/internal/ui-ux-spec-pamphlet-workbench.md`.

## Intent

This variant explores maximum screen economy while preserving the same shape and function.

Its personality is:

- compressed
- severe
- efficient
- almost editorial

It should feel like a dense professional instrument where no element is given ceremonial space.

## Fixed shape

Render the same fixed workbench shape:

- shallow top strip
- left upstream sliver
- central two-page spread
- far-right audit ledger sliver

Do not alter the fundamental panel ownership.

## Compression goals

- make the top strip thinner than Proposal A
- make the slivers feel tighter and more information-dense
- reduce whitespace and header depth
- use compact ribbons and segmented controls
- keep tables dense but still readable

The key experiment is how far screen economy can be pushed before clarity suffers.

## Surface behavior

### Top strip

Use a very thin command band.

Show:

- source
- workflow
- run / stop
- activity

Avoid generous spacing.

### Left upstream sliver

Show:

- only the most essential upstream metrics
- collapsed file tree affordance at bottom

Treat it like a narrow binding margin with utility, not a side panel.

### Primary Surface

Show:

- very dense table-first inspection layout
- concise headers
- compact filters
- checkboxes and sort affordances

### Secondary Surface

Show:

- compact preview mode toggle
- compact consequence render
- compact but explicit mutation controls

### Audit ledger sliver

Keep it extremely compressed:

- tiny counts
- terse labels
- obvious expand affordance

## Workflows to include

Same four workflows as Proposal A:

- Normalize
- Delete Junk
- Repair Defaults
- Dashboard

## Visual treatment

Keep it stripped down:

- low whitespace
- thin borders
- compact rows
- compressed labels
- no decorative flourishes

Use schematic visual language only.

## What should vary in this proposal

Compared to the other three proposals, this one should:

- be the most aggressive about density
- feel like the strongest statement of screen economy
- test whether compactness remains humane and navigable
